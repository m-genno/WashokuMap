# WashokuMap アーキテクチャ設計書

> 和食店に特化したインバウンド向け Web アプリケーション
> Status: Draft / 最終更新: 2026-06-21

---

## 1. プロダクト概要

訪日外国人(インバウンド)を主対象に、**食べログ・一休などに未掲載の和食店**を発掘・検索・予約できる Web アプリ。

### コアコンセプト
| 原則 | 設計上の意味 |
|---|---|
| インストール不要 | ネイティブアプリを作らず **PWA(Webアプリ)** で提供 |
| 登録不要 | 匿名で全機能を体験でき、**ログインは任意のアップグレード** |
| 言語の壁を越える | 予約・口コミを**自動翻訳**して店舗と客の橋渡しをする |
| 人手前提の運用 | 非掲載店の開拓・予約仲介は**予約デスク(人間)**が介在する前提で設計 |

### スコープ(MVP)
- 自由語句による和食店検索(店名・ジャンル・エリア)
- 地図 + 一覧の二面表示、店舗詳細
- リクエスト型 Web 予約(在庫管理なし)
- 検索履歴・お気に入り(匿名=ブラウザ保存、ログイン=サーバ同期)
- 口コミ・評価の閲覧/投稿

### 非スコープ(将来)
- リアルタイム空席在庫管理
- 決済・前金徴収
- ネイティブアプリ配布
- 店舗向け管理ダッシュボードのフル機能化(MVPは最小限)

## 3. 技術スタック

| レイヤ | 採用候補 | 選定理由 |
|---|---|---|
| フロント | **Next.js (React) + TypeScript** | SSR/SEO に強くPWA化が容易。検索結果のSEOはインバウンド集客に直結 |
| PWA | Service Worker / Web App Manifest | インストール不要のまま「ホーム追加」「プッシュ通知」を実現 |
| 地図 | **Google Maps Platform** または Mapbox | 外国人に馴染み深いGoogle Maps。コスト次第でMapboxも検討 |
| バック | Next.js Route Handlers / Node | フロントと一体運用しMVPを素早く立ち上げ |
| DB | **PostgreSQL + PostGIS** | 全文検索・地理検索・リレーションを単一DBで完結 |
| 検索 | まず Postgres FTS、規模拡大で **Meilisearch / Algolia** | 多言語・タイポ許容・高速サジェストが必要になったら移行 |
| 認証 | **Auth.js (NextAuth)** + 匿名トークン | Google/Apple OAuth を標準サポート、匿名併用が可能 |
| 翻訳 | DeepL API / Google Translate API | 予約・口コミの双方向翻訳 |
| 通知 | Resend or SendGrid(メール)/ Twilio(SMS)/ Web Push | 予約確定通知。店舗への一次連絡はメール+人手 |
| ホスティング | Vercel など | Next.js と親和性が高い。DBはマネージド(Supabase/Neon/RDS) |

> 補足: Supabase を採用すると PostgreSQL + Auth + Storage + 全文検索をまとめて入手でき、MVP の立ち上げが速い。本書のDB/認証はSupabaseでも実現可能。

## 4. 認証・ユーザーデータ設計

### 方針: プログレッシブ・アイデンティティ
登録不要の体験を壊さず、希望者だけサーバ同期へ「昇格」させる。

```
[初回訪問]
   │  端末に anonymous_id(UUID) を発行 → localStorage 保存
   ▼
[匿名利用] ─ 検索履歴/お気に入り は localStorage に保存(サーバ任意ミラー)
   │         予約時の氏名・連絡先は localStorage に保存し次回オートフィル
   ▼
[任意サインイン] ─ Google / Apple ワンタップ
   │  サインイン時に anonymous_id のデータを user_id へマージ
   ▼
[ログイン利用] ─ 複数端末で同期、予約履歴・お気に入りを永続化
```

### データ保存場所の使い分け

| データ | 匿名時 | ログイン時 |
|---|---|---|
| 検索履歴 | localStorage(主) | サーバ(端末間同期) |
| お気に入り | localStorage(主) | サーバ |
| 予約時の個人情報(氏名/メール/電話) | localStorage(オートフィル用) | サーバ(プロフィール) |
| 予約リクエスト本体 | サーバ(必須・匿名でも保存) | サーバ |
| 口コミ投稿 | サーバ(投稿には軽い本人性が必要、後述) | サーバ |

### 重要な論点
- **匿名でも予約はサーバに保存する**: 予約は店舗・予約デスクと共有する業務データなのでブラウザ保存では成立しない。匿名予約は `anonymous_id` + 入力連絡先で紐付ける。
- **個人情報をサーバに持つ責任**: プライバシーポリシー、個人情報保護法(APPI)、EU客向けGDPRへの対応が必要。匿名localStorage中心の機能ほどこの負担が軽い。
- **ソーシャルログインの選定**: 訪日客の保有率から **Google + Apple** を採用。LINE は日本/台湾/タイ偏重のため主軸にしない(将来オプション)。
- **口コミの本人性**: スパム防止のため、口コミ投稿は「ログイン必須」または「予約実績のある匿名ユーザのみ」に制限する設計を推奨(詳細は §6)。

## 5. Web予約設計

### 基本思想: 在庫管理ではなく「リクエスト転送」
主ターゲットの非掲載店はオンライン空席管理を持たない。予約を**「在庫の確定処理」ではなく「メッセージのやり取り」**としてモデル化する。

### 店舗を2タイプに分類

```
                ┌─ 予約方式を判定 ─┐
                ▼                  ▼
   [Aタイプ: 既存予約システム店]   [Bタイプ: 非掲載店 = 主ターゲット]
   食べログ/一休/OpenTable等        オンライン予約手段なし
        │                              │
   外部予約ページへ                リクエスト型予約
   ディープリンク or 連携API          (本アプリで完結)
```

### リクエスト型予約フロー(Bタイプ)

```
ユーザ
  │  ① 予約リクエスト送信
  │     希望日時 / 人数 / 氏名 / 連絡先 / 要望(アレルギー等) / 言語
  ▼
アプリサーバ  reservation を status=requested で作成
  │  ② 内容を日本語へ翻訳して店舗へ通知(メール/SMS)
  │     ＋予約デスク(人手)へタスク通知
  ▼
店舗 / 予約デスク  ③ 可否を回答(店が電話/メール、またはデスクが代行確認)
  │
  ▼
アプリサーバ  status=confirmed / declined / counter_offer に更新
  │  ④ 結果を客の言語へ翻訳して通知(メール/Push)
  ▼
ユーザ  ⑤ 確定 or 代替案受領
```

### 予約ステータス(状態遷移)
```
requested ──▶ confirmed ──▶ completed
   │  │  └──▶ counter_offer ──▶ confirmed / cancelled
   │  └─────▶ declined
   └────────▶ cancelled        (no_show は完了後に別途記録)
```

### インバウンドの差別化
- リクエスト内容を**自動で日本語化して店へ**、回答を**客の言語へ翻訳して返す**。
- **翻訳文に加えて、客が入力した原文もそのまま併記して送る**。誤訳・ニュアンス欠落の保険になり、店側が固有名詞や元の表現を確認できる(回答を客へ返す際も同様に、店の原文＋客言語訳を併記)。
  - 例: 店向けメールは「【日本語訳】… / 【原文(English)】…」の2ブロック構成。
  - DB上も翻訳文と原文の双方を保持する(予約は `requests`＝原文 / `requests_ja`＝訳文。口コミは `body`＝原文 / `body_translations`＝訳文)。
- アレルギー/宗教上の制限/予算を**定型項目化**し、言語に依存せず伝達。
- 補助として `tel:` ワンタップ電話も用意(言語の壁があるためフォームが本命)。

### Aタイプの扱い
- `reservation_url` を保持し、店舗詳細に「公式サイトで予約」ボタンを表示してディープリンク。
- 将来 TableCheck / トレタ / ebica 等のパートナーAPIに対応すれば、アプリ内完結も可能。

## 6. 口コミ・評価設計(メモ追記要望)

| 論点 | 方針 |
|---|---|
| 投稿資格 | **当該店で予約実績(確定/完了)があるユーザに限定。匿名ユーザも実績があれば投稿可**(決定事項) |
| 評価軸 | 総合星(1〜5)。将来「料理/接客/雰囲気/コスパ」の多軸へ拡張可能 |
| 多言語 | 投稿言語を保存し、閲覧者の言語へ**オンデマンド翻訳**(原文も保持) |
| モデレーション | 公開前 or 公開後の通報ベース審査。`status` で管理 |
| 集計 | 店舗に `rating_avg` / `rating_count` をキャッシュ列として保持し検索高速化 |

## 7. 主要画面とAPI

### 画面
1. **検索/結果画面** — 検索バー、地図(マーカー)、一覧、絞り込み
2. **店舗詳細画面** — 写真、基本情報、地図、電話、予約ボタン、口コミ
3. **予約フォーム** — 日時/人数/連絡先(オートフィル)/要望
4. **マイページ(軽量)** — お気に入り、検索履歴、予約状況
5. **口コミ投稿**

### 代表的なAPIエンドポイント(REST例)
```
GET  /api/restaurants/search?q=&lat=&lng=&radius=&genre=   店舗検索(全文+地理)
GET  /api/restaurants/{id}                                 店舗詳細
POST /api/reservations                                     予約リクエスト作成
GET  /api/reservations/{id}                                予約状況参照
POST /api/reservations/{id}/cancel                         予約キャンセル
GET  /api/restaurants/{id}/reviews                         口コミ一覧
POST /api/restaurants/{id}/reviews                         口コミ投稿
GET  /api/me/favorites  / POST  / DELETE                   お気に入り
POST /api/auth/merge-anonymous                             匿名→ログイン統合
```

## 8. 横断的関心事

| 項目 | 方針 |
|---|---|
| 多言語 (i18n) | UI は en/ja/zh-Hans/zh-Hant/ko など。コンテンツは翻訳キャッシュ |
| SEO | 店舗詳細をSSRで多言語生成 — 検索流入はインバウンド集客の生命線 |
| プライバシー | 個人情報の保存に同意取得、ポリシー整備、APPI/GDPR考慮 |
| アクセシビリティ | モバイルファースト、低速回線(海外SIM/ローミング)を想定し軽量化 |
| 計測 | 検索→詳細→予約のファネル分析。非掲載店開拓の優先順位付けに活用 |
| 不正対策 | 予約スパム/いたずら防止にレート制限・メール/SMS確認 |

## 9. 店舗データ投入(決定事項)

非掲載店の開拓は人手で行うため、**人手登録UI と CSVインポートの両方**を用意する。

### 9.1 人手登録UI(運用者向け管理画面)
- 営業担当・予約デスクが1店舗ずつ入力/編集する内部用フォーム。
- 入力項目: 店名(多言語)、住所、電話、ジャンル、営業時間、写真、`listing_type`、`reservation_mode`、`reservation_url` 等。
- 住所入力時に**ジオコーディング**して `location`(緯度経度)を自動補完。
- 認可: 運用者ロール(`staff` / `admin`)のみアクセス可。一般ユーザUIとは分離。
- ステータス管理: `draft` で下書き保存 →内容確認後 `published` で公開。

### 9.2 CSVインポート(一括投入)
- 既存の店舗リスト(営業先リスト等)をまとめて取り込む。
- フロー: **CSVアップロード → バリデーション/プレビュー → 取り込み実行**。
  - 必須列(店名/住所 等)の欠落・型エラーを事前に検知して表示。
  - 住所からバッチでジオコーディングし `location` を補完。失敗行はエラーレポート出力。
  - 既存店との重複判定キー(例: 電話番号 or 店名+住所)で**新規/更新**を判定(upsert)。
- 取り込み結果は `import_batch` として記録し、失敗行のみ再投入できるようにする。
- インポート直後は `draft` 取り込み →人手で確認後に一括 `published` も可能にする。

> 投入経路の違いは `restaurant.source`(`manual` / `csv` / 将来 `api`)で区別し、出所を追跡する。
