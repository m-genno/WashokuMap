# WashokuMap 管理者向け操作ガイド

運用者(店舗データの登録・公開、予約デスク、口コミ対応など)向けの操作手順をまとめます。
画面はすべて `/admin` 以下にあり、日本語表示です。お客様向けの公開画面(`/`, `/search`, `/restaurants/...`)は多言語対応です。

- 対象読者: 店舗データ運用・予約デスク・モデレーション担当
- 関連ドキュメント: [architecture.md](./architecture.md) / [data-model.md](./data-model.md)

---

## 1. ログインと認証

本アプリにはログイン画面はなく、**運用者トークン(`ADMIN_TOKEN`)** で管理APIを保護します。

1. 各管理画面の上部にある「運用者トークン」欄にトークンを入力します。
   - 入力値は**この端末のセッション(sessionStorage)**にのみ保存され、サーバには保存されません。
   - ブラウザのタブを閉じると消えます。再度入力してください。
2. `ADMIN_TOKEN` が**未設定の開発環境**では、トークンは空のままで操作できます(API は開放)。
   - 起動ログに `[admin] ADMIN_TOKEN is not set; admin API is OPEN (dev only).` の警告が出ます。
3. **本番では必ず `ADMIN_TOKEN` を設定**してください(→ [9. 環境変数](#9-環境変数運用設定))。

> セキュリティ注意: トークンは共有シークレットです。操作ログ(監査)には「トークンのハッシュ(`admin:xxxxxxxx`)」とIPが残ります。個人を特定したい場合は運用者ごとにトークンを分けてください。

## 2. 管理トップ `/admin`

各機能への入口です。

| メニュー | 役割 |
|---|---|
| 予約デスク | 予約リクエストの確認・確定/お断り/完了など |
| 口コミ モデレーション | 通報された口コミの確認・非表示/再公開 |
| 店舗一覧・公開 | 登録済み店舗の検索・編集・公開 |
| 店舗を登録 | 1店舗ずつ手入力で登録 |
| CSVインポート | 営業先リスト等の一括投入 |
| 操作ログ | 管理操作の履歴(差分つき) |
| メンテナンス | 孤立アップロード画像の掃除 |

## 3. 店舗の登録・編集・公開

### 3.1 状態(ステータス)
店舗には3つの状態があります。

- **下書き(draft)**: 非公開。検索・詳細に出ません。投入直後の既定。
- **公開中(published)**: お客様の検索・詳細に表示されます。
- **休止(closed)**: 非公開(一時休業など)。

> **公開には位置情報が必須**です。住所からジオコーディングできなかった店舗は、緯度・経度を手入力するまで公開できません(公開ボタンが無効、編集時は `位置情報がないため公開できません` を返します)。

### 3.2 1店舗ずつ登録 `/admin/restaurants/new`
- 必須は **店名** のみ。住所を入れると自動でジオコーディングして地図に表示します。
- 予約方式は3種:
  - **リクエスト予約(request)**: アプリ内でお客様が予約リクエスト → 予約デスクで可否対応。
  - **外部サイト(external)**: 予約は外部URLへ誘導。`外部予約URL` を入力。
  - **電話のみ(phone_only)**: 電話番号への発信のみ。
- ジャンル(複数可)、価格帯(¥〜¥¥¥¥)、紹介文、店名(英語)を任意で入力。
- 公開状態は登録時に選べます(既定は下書き)。

### 3.3 一覧・検索・状態変更 `/admin/restaurants`
- 上部の検索ボックスで **店名・住所・英名・多言語名・紹介文** を部分一致検索(日本語の途中文字でもヒット)。
- 状態タブ(下書き / 公開中 / 休止 / すべて)と**併用**できます。
- 各行の操作:
  - **編集** → 編集画面へ
  - **公開する**(位置情報があるときのみ有効)/ **下書きに戻す** / **休止**
  - 公開中は「公開ページを見る ↗」で実際の表示を確認

### 3.4 編集 `/admin/restaurants/<id>/edit`
全項目を更新できます。

- **基本情報**: 店名 / 英名 / 住所 / 電話 / Webサイト / 予約方式 / 価格帯 / 紹介文 / 公開状態。
  - 英名は `name_translations.en` のみ更新し、他言語訳(ja/zh/ko 等)は保持します。
- **緯度・経度(任意)**: 入力すると住所より優先。**空欄にして保存すると住所から再取得**します。
  - フォームは現在の座標を初期表示するため、通常は保存しても位置はそのまま保持されます。
- **ジャンル**: チェックで総入れ替え。
- **写真**:
  - 「画像をアップロード」: 端末の画像をアップロード。サーバで**自動的に軽量化(最大1600px/WebP)+ サムネイル(最大400px)生成**(「軽量化済」表示)。
  - 「+ URLで追加」: 外部画像URLを貼り付け(サムネは作られません)。
  - 「メイン」ラジオで主写真を1枚指定。
- **営業時間**: 曜日 + 開店/閉店 + 備考の行を追加。**1日に複数行(昼/夜)**を登録できます。

> 編集内容は**操作ログに差分(変更前→変更後)**として記録されます(→ [7. 操作ログ](#7-操作ログ監査)。

## 4. CSVインポート `/admin/restaurants/import`

営業先リスト等を一括投入します。**投入された店舗は下書き**になり、内容確認後に公開します。

### 4.1 CSV 仕様
- 1行目はヘッダ。利用できる列:

  | 列 | 必須 | 内容 |
  |---|---|---|
  | `name` | ◯ | 店名 |
  | `name_en` | | 英語名 |
  | `address` | 推奨 | 住所(ジオコーディングで地図表示) |
  | `phone` | | 電話番号 |
  | `website_url` | | Webサイト |
  | `genres` | | ジャンルコードを `;` 区切り(例 `sushi;izakaya`) |
  | `reservation_mode` | | `request` / `external` / `phone_only`(既定 `request`) |
  | `reservation_url` | | 外部予約URL |
  | `price_range` | | 1〜4 |

- ジャンルコード: `sushi, tempura, soba, udon, unagi, yakitori, tonkatsu, ramen, izakaya, kaiseki, teppanyaki, shabushabu, sukiyaki, donburi, washoku`
- サンプル: `db/seed/sample_restaurants.csv`

### 4.2 重複の扱い
同一店舗の判定は **電話番号 → なければ 店名+住所** の順。一致すれば**更新**、なければ**新規**として扱います。

### 4.3 取込の流れ
1. 「CSVファイルを選択」で読み込み(ロード時に自動でダイアログは開きません)。
2. **取込前プレビュー**が自動表示されます。各行を **新規 / 更新 / 失敗** で判定:
   - 更新行は上書き対象の既存店名を表示。
   - 失敗行はエラー理由(例: `name は必須です`)を表示。
   - ボタンは「**N 件をインポート(新規x/更新y)**」と取込可能件数を表示(失敗は除外)。
3. インポート実行後、件数結果に加え、**取り込んだ店舗をその場で一覧表示**。検索/編集と同じ操作で、確認・編集・公開までこの画面で行えます。

## 5. 予約デスク `/admin/reservations`

リクエスト予約(`reservation_mode='request'`)のお客様からの予約を処理します。

### 5.1 検索・絞り込み
- **検索**: お客様名 / 店名 / メール / 電話 の部分一致。
- **希望日(から/まで)**: `desired_at` の範囲(「まで」はその日を含む)。
- **状態タブ**: 未対応 / 確定 / 完了 / すべて。これらは併用できます。

### 5.2 状態と操作
予約は次の状態を持ち、許可された遷移のみ実行できます(監査ログに記録)。

| 現在 | 押せる操作(遷移先) |
|---|---|
| 未対応(requested) | 確定する / 代替を提案(counter_offer)/ お断り(declined)/ キャンセル |
| 代替提案(counter_offer) | 確定する / お断り / キャンセル |
| 確定(confirmed) | 来店済み(completed)/ No-show / キャンセル |
| お断り・キャンセル・完了・No-show | (終端: 変更不可) |

各予約には希望日時・代替希望・人数・予算・連絡先・利用言語・要望(原文+日本語訳)・食事制限が表示されます。

### 5.3 お客様への自動通知
状態を **確定 / お断り / 代替提案 / キャンセル** にすると、**お客様の言語(guest_lang)で自動メール**を送信します(定型テンプレート、店舗名もお客様の言語で表示)。

- メール送信には `RESEND_API_KEY` と `NOTIFICATION_FROM_EMAIL` が必要。未設定時は送信せずログ出力にフォールバックします(操作は成立)。
- `APP_BASE_URL` を設定すると、メールに**予約状況ページ**へのリンクを掲載します。
- お客様は予約状況ページ `/reservations/<id>` で最新状態を確認でき、**代替提案には「承諾(→確定)/お断り(→キャンセル)」**をその場で回答できます。

> 新規リクエスト時には、`RESERVATION_DESK_EMAIL` 宛に「予約デスク向け通知」(要望の日本語訳併記)も送られます。

## 6. 口コミ モデレーション `/admin/reviews`

通報された口コミや非表示中の口コミを確認し、対応します。

- フィルタ: **通報あり** / **非表示中** / すべて。
- 各口コミに評価・本文(原文)・**日本語訳**・**通報理由**・**添付写真**が表示されます。
- 操作:
  - **非表示にする(hidden)**: 公開から外します。
  - **再公開する(published)**: 公開に戻します。
- 非表示にすると、その店舗の**評価平均・件数が自動再計算**されます(非表示分は集計から除外)。
- 口コミの通報はお客様が公開ページから行えます(資格不要)。投稿自体は**予約実績(確定/来店済み)のあるお客様のみ**可能です。

## 7. 操作ログ(監査)`/admin/audit`

管理操作の履歴を確認できます。

- 記録対象: 店舗の登録・編集・状態変更、CSV取込、予約対応、口コミ対応、画像整理。
- 各エントリに **操作種別・要約・操作者(トークンのハッシュ)・IP・日時** を表示。
- **店舗編集**は変更フィールドの**差分(項目: 変更前 → 変更後)**を表示します。
- アクション種別で絞り込み可能。

---

## 8. メンテナンス: 孤立画像の掃除 `/admin/maintenance`

どの口コミ・店舗にも紐付かないアップロード画像(投稿せず破棄、編集での差し替えで発生)を削除します。

1. **猶予(時間)** を指定(既定24h)。**これより新しい未参照ファイルは保持**します(投稿直前のアップロードを誤削除しないため)。
2. 「**確認(削除しない)**」でドライラン。対象件数・解放見込み容量を表示。
3. 「**削除する**」で実削除(確認ダイアログあり)。実削除は操作ログに記録されます。

> 管理対象のファイル名パターン以外には触れません。定期実行したい場合は `POST /api/admin/uploads/cleanup`(`{"dryRun":false}`)を cron 等で叩けます。

## 9. 環境変数(運用設定)

`.env.local`(ローカル)や本番の環境変数で設定します。雛形は `.env.example`。

| 変数 | 用途 | 未設定時 |
|---|---|---|
| `DATABASE_URL` | PostgreSQL 接続先 | 必須 |
| `ADMIN_TOKEN` | 管理API保護トークン | **未設定だと管理APIが開放(本番では必ず設定)** |
| `RESEND_API_KEY` | メール送信(Resend) | メールは送らずログ出力 |
| `NOTIFICATION_FROM_EMAIL` | 送信元メールアドレス | 同上 |
| `RESERVATION_DESK_EMAIL` | 新規予約のデスク宛先 | デスク通知なし |
| `APP_BASE_URL` | 公開URL(オリジン) | お客様メールに状況ページURLを載せない |
| `DEEPL_API_KEY` | 要望・口コミの日本語訳 | 翻訳なし(原文のみ) |
| `GEOCODE_API_URL` | 住所→緯度経度 | 既定の Nominatim(OSM) |
| `UPLOAD_DIR` | アップロード画像の保存先 | `./uploads` |

> 注意: `.env*` は Git 管理外です(`.env.example` のみコミット)。本番では永続化が必要なアップロードは `UPLOAD_DIR` を永続ボリュームに(将来はオブジェクトストレージ化)してください。

## 10. テストデータ(開発・動作確認用)

```bash
npm run db:up      # DB起動(スキーマ自動適用)
npm run db:seed    # サンプル + 約200店舗 + 関連データ(口コミ/予約/通報/営業時間)を投入
npm run dev        # http://localhost:3000
```

- `db:seed` は再実行しても安全(冪等)。
- バルク投入分だけ削除: `delete from restaurant where id::text like 'bbbb0000-%';`
- 詳細は `db/seed/bulk_restaurants.sql` / `db/seed/bulk_activity.sql` を参照。

## 11. よくあるトラブル

| 症状 | 確認・対処 |
|---|---|
| 管理画面で「認証が必要です」 | 上部の運用者トークンが正しいか確認。本番は `ADMIN_TOKEN` と一致が必要。 |
| 店舗を公開できない | 位置情報なし。住所を見直すか、編集画面で緯度・経度を手入力。 |
| お客様にメールが届かない | `RESEND_API_KEY` / `NOTIFICATION_FROM_EMAIL` を確認。未設定時はログ出力のみ。 |
| 要望・口コミの日本語訳が出ない | `DEEPL_API_KEY` 未設定。原文のみ保存されます。 |
| 予約状態のボタンが押せない | 終端状態(お断り/キャンセル/完了/No-show)からは変更できません。 |
| アップロード画像が表示されない | 本番で `UPLOAD_DIR` が永続化されているか確認(コンテナ再作成で消えることがあります)。 |
